fury.js

Four methods to freedom.    

Sigh.

If you're anything like me, you worked in the great world of Flex. We had great frameworks, like Swiz and Robotlegs. We had an IDE that was as good as its Eclipse container allowed it to be. We had an awesome community.

And then Adobe went and screwed it all up. Now you're likely looking at JavaScript, and it's not pretty. That's why I cobbled together this little toolkit.

fury.js, combined with tools like Knockout, is how I'm writing JavaScript-based apps using the same architectures that I used with Swiz: controllers, presentation models, service delegates, injections, bindings. They're all there with just a little help from fury.js.

Here's a quick guide, just for you folks who speak MXML, aimed mostly at the Swiz dialect.

[Inject]

Ahh, [Inject]. You asked, and your bean was delivered. fury.js has an equivalent, and you don't need to worry about wacky "Oops, I called inject() on instantiation, but I hadn't registered the dependency yet!" kind of stuff. fury.js handles late-binding of dependencies just fine.

Where you would write:

[Inject] public var service : SomeService

fury.js lets you write:

fury.inject( this, "someService", someService )

Inside that class that is "this," you'll get a property named "someService" that is the bean registered as "someService." This should sound awfully familiar.

Going further, you can have something happen right after injection takes place:

fury.inject( this, "someService", someService, function() {
    console.log( "someService was injected into:" );
    console.log( this ); // note that "this" is resolved correctly!
}

[EventHandler]

The core of it all, letting you escape the tree that was the displaylist and enter the realm of good MVC. A lot of JavaScript I see doesn't acknowledge this yet, so fury.js provides a central event dispatcher much like really early Swiz (later replaced by the application itself!). Whereas most other JavaScript "pub/sub" frameworks rely on global-level event handlers, fury.js assumes your event handlers are located within an instance a freestanding, independent instance of a class (e.g. a controller).

If that made no sense, let me put it this way: most JavaScript event frameworks act like you defined your handlers within your top-level MXML application file, and then you dug into your entire view tree to take action.

Where you would write:

[EventHandler( event="GreetingEvent.SAY_HI", properties="message") ]
public function sayHi( message : String ) : void
{
    // stuff
}

fury.js lets you write:

fury.subscribe( "greeting.sayHi", this, function( message ) {
    // stuff
});

dispatchEvent( event )

Back in 2000 or so, Flash's event system opened my eyes to a new world of development. It led to me writing Model-Glue, which really jumpstarted my career. It's the core of how we build good Flex applications, so I had to have it in JavaScript-land.

Where you would write:

dispatcher.dispatchEvent( new GreetingEvent( GreetingEvent.SAY_HI, "Hi mom!" ) );

fury.js lets you write:

fury.publish( "greeting.sayHi", "Hi Mom!" );

That second argument can be anything, so you can do fun things like:

fury.publish( "contact.save", { firstName : "Mark", lastName: "Weir"} );

Or, if you choose to use the "M" part of "MVC":

fury.publish( "contact.save", new Contact( 'Steve', 'Peat' ) );

BeanProvider.as

Yeah, it was really nice to define beans in MXML and all that good stuff. Nothing quite that elegant exists in JavaScript (yet), so fury.js provides a simple way to register a bunch of beans at once.

Where you would write:

<service:ContactService />
<control:ContactController />
<pm:ContactEditorPresentationModel />

fury.js lets you write:

fury.register({
    contactService : new ContactService(),
    contactController : new ContactController(),
    contactEditorPresentationModel : new ContactEditorPresentationModel(),
})
